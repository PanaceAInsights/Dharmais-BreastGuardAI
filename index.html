BreastGuard AI | Data Requirements Specification

RS Dharmais National Cancer Center Collaboration
Version 1.0 Prepared by: Dr Yagiz Alp Aksoy MD PhD, PanaceAI Date: 23 February 2026

Overview

Purpose: This document describes the data fields we would like to use to develop, train, and validate BreastGuard AI: a machine learning model that predicts breast cancer risk stratification, survival, and recurrence tailored to the Indonesian population.

Approach: We use ensemble machine learning (XGBoost, Random Forest, SVM) with SHAP explainability. This is the same methodology used in our published and peer-reviewed tools for colorectal polyps (ANZ J Surgery, 2025), medullary thyroid carcinoma (Histopathology, 2024), and mesothelioma (Pathology, 2023).

Deliverable: A de-identified, tabular dataset (Excel/CSV), one row per patient, with the fields described in this document.

Where we are: Based on our previous Q&A, we understand there are approximately 2,125 breast cancer cases (2022) available with clinical data, pathology reports, IHC markers, and digitised slides (Philips SG300, 40x). This is an excellent foundation to work from.
Flexibility note: The tables in each tab describe our ideal dataset. We understand that not every variable may be available or easy to extract, and that is perfectly fine. We are happy to work with whatever data can be provided. Even a subset of these variables across the cohort will allow us to build a meaningful model. The "Usefulness" column simply indicates which variables tend to have the strongest predictive value based on our experience with similar models.

5. Outcome Data (The Target Variables)

Why outcomes matter most: Outcome data is what allows the AI model to learn patterns between patient characteristics and what happens to them. Without survival status and follow-up duration, we can still build a descriptive tool, but a predictive model needs outcomes to train on. Even partial outcome data (e.g., survival status known for 70% of patients) is very workable.

7. Digital Pathology (Phase 2)

Confirmed Infrastructure

Scanner Details

  • Scanner: Philips SG300
  • Resolution: 40x equivalent
  • Scan time: ~62 sec/slide (total)
  • Throughput: ~30 slides/hour
  • Avg file size: ~1.3 GB per slide

Annotation & Scoring

  • Annotations: Available via IMS viewer
  • Pathologist scoring: Yes (WHO criteria)
  • Tumour regions: Can be marked
  • Unstained slides: Available if IHC needed in Australia
Note: Digital pathology integration is a Phase 2 enhancement. For the initial model, we only need the tabular clinical + pathology data described in the other tabs. Slide images can be incorporated later for deep learning-based tumour grading and feature extraction.

8. Data Delivery Specification

Format

  • Excel (.xlsx) or CSV (.csv)
  • One row per patient
  • One column per variable
  • English column headers preferred
  • Consistent coding (no mixed formats)

Governance & Security

  • Fully de-identified before transfer
  • Encrypted transfer (secure link or encrypted email)
  • Ethics approval / data sharing agreement in place
  • Compliant with Indonesian PDPA and Australian Privacy Act
